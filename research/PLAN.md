## 逐笔通道核心结论 — 成交 + 委托 混合在同一通道 ✅

**重要更新 (经字节级验证)**: 9/5803 和 39/3901 **都是"逐笔行情"混合通道**, 即
**逐笔成交 + 逐笔委托 在同一 template_id 下混播**。不需要"两个通道",
SSE Binary 协议把这两类事件通过记录内的 **1 字节 action 字符** 做区分。

### 事件类型区分 (关键发现)

每条 FAST 记录中, 在 SecurityID 之后有一个 1 byte 字符字段作为 **Action** 标记:

| 字节   | ASCII | 事件                       |
|--------|-------|----------------------------|
| `0x54` | `'T'` | **逐笔成交 Trade**         |
| `0x41` | `'A'` | **逐笔委托 - Add 新增**    |
| `0x44` | `'D'` | **逐笔委托 - Delete 撤单** |
| `0x43` | `'C'` | **Cancel**                |

ApplSeqNum 在通道内**对 'T' 和 'A' 严格 +1 递增**; 'D'/'C' 的 4B 字段是**引用**的旧 OrderApplSeqNum。

## 通用流程 (已执行完 12/12 通道)

1. **抽样**: `build/split_frames data/output.pcap 5261 <type>,<sub> 6 > tmp/ch_<type>_<sub>.txt`
2. **查首字节特征** 确定协议族 (FIX / M201 / FAST)
3. **按族解码**:
   - FIX → SOH 切 tag + 查 tag 语义
   - M201 → 对齐 49B 包头 + NoEntries × SecurityEntry (69B + 19B × NoMDEntries)
   - FAST → 按 stop-bit 取字段, 对齐 template_id, 区分 PMAP
4. **识别市场段**: 看 SecurityID 前缀、Symbol、MDStreamID、TradingPhase 代码
