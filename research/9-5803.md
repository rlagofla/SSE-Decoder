# 通道 (9, 5803) 分析 — **沪市 股票/ETF 逐笔行情 (成交 + 委托 混合)**

## 1) 最终结论 (经 CSV 交叉验证 99.978%)

本通道 = **UA5803 (官方文档 2.0.8 版之后新增的合并模板)**，单条 FAST 消息里同时承载:

| Action 字符 | ASCII | 含义            | 占比   |
|-------------|-------|-----------------|--------|
| `'T'`       | 0x54  | 逐笔成交 Trade  | 39.4%  |
| `'A'`       | 0x41  | 逐笔委托 新增   | 42.2%  |
| `'D'`       | 0x44  | 逐笔委托 撤单   | 18.4%  |
| `'C'`       | 0x43  | 取消            | <0.1%  |

**验证指标**: 全量 pcap 解码 246,105 条记录和通联 CSV `mdl_4_24_0.csv` 按 (Channel, BizIndex) 做 join, SecurityID / Type / BSFlag / Price / Qty / TradeMoney / BuyOrderNO / SellOrderNO / TickTime 全部匹配的比例:
- 'A' 99.987%, 'D' 99.982%, 'T' 99.967%
- 总体 **99.978% (246,051 / 246,105)**

未匹配部分来源于"字段值 = 5803 的 FAST int 字节恰好是 `2d ab`"造成的伪 TID 命中 — 用 PMAP MSB + 最短间隔 14B 的启发式过滤后剩 0.02%, 要彻底消除需改用完整 FAST 字典式解码。

## 3) body 解压后的线缆布局

```
PMAP (1B, MSB=1 停止)
TID  (FAST u32 = 5803, 字节固定为 2d ab)
[BizIndex]    (FAST u64 mandatory, "业务序列号", 按 Channel +1)
[Channel]     (FAST u32 mandatory)
[SecurityID]  (6B ASCII, 前 5 字节 '0'..'9' MSB=0, 第 6 字节 MSB=1)
[TickTime]    (FAST u32 nullable, HHMMSSXX × 100, e.g. 9535063 = 09:53:50.63)
[Action]      (FAST char, 'T'/'A'/'D'/'C')
中间 FAST 整数序列 (随 Action 变化, 见第 5 节)
BSFlag        (FAST char, 'B' / 'S' / 'N', 字节值 0xC2 / 0xD3 / 0xCE)
[ExtCode]     (1B RAW, 取值 0x40~0x4F, 语义可能是分类 tag; 部分短记录省略)
```

方括号 `[...]` 表示可被 PMAP 位图 / FAST 操作符省略。

## 4) **关键坑 1: FAST nullable 编码 (+1 偏移)**

FAST 把 u32/u64 字段分为 mandatory 和 optional (nullable)。**nullable 字段的线缆值 V 和业务值 X 不直接相等**:

```
V == 0x80  (单字节 "编码=0, stop 置位")   →   X = NULL
V >  0                                    →   X = V − 1
```

对照本通道各字段:

| 字段         | nullable? | 语义                                           |
|--------------|-----------|------------------------------------------------|
| BizIndex     | 否        | 线缆值就是业务值                               |
| Channel      | 否        | 同上                                           |
| TickTime     | **是**    | 线缆值 − 1 = 业务值 (9535064 → 9535063 = 09:53:50.63) |
| OrderNO      | **是**    | 6000458 → 6000457                              |
| Price        | **是**    | 6331 → 6330 → 6.330 元 (×10³)                  |
| Qty          | **是**    | 1100001 → 1100000 → 1100 股 (×10³)             |
| TradeMoney   | **是**    | 'T': 7371634001 → 73716.340 元 (×10⁵); 'A'/'D': ×10³ |

**这是我第一版分析全程错位的根源。**之前以为 pcap 解出来的值"凭空大 1", 其实是 FAST 规定的编码规则。

## 5) **关键坑 2: 精度不同 — 'A'/'D' 的 Money 是 ×10³, 'T' 的 Money 是 ×10⁵**

STEP 层显示的小数位数不同:
- 逐笔成交 `TradeMoney` = `"73716.34000"` (5 位)
- 逐笔委托的第 4 个 int (在 CSV 里也叫 TradeMoney, 实际是委托金额指示) = `"5900.000"` (3 位)

所以 FAST int 解码后要按 Action 区分缩放倍数:
- `'T'`:            `X / 10⁵` → 元
- `'A'/'D'/'C'`:    `X / 10³` → 元

## 6) **关键坑 3: PMAP 省略导致字段前导数量可变**

帧内记录有三种前导字段组合 (观测):

| prefix 条数 | 含义                                 | 发生场景             |
|-------------|--------------------------------------|----------------------|
| 3           | [BizIndex, Channel, TickTime] 全显式 | **每帧第一条**       |
| 1           | [TickTime] 单独出现                  | 帧中 TickTime 变化时 |
| 0           | 全部继承                             | 最常见 (~99%)        |

对应操作符:
- **BizIndex** (increment): 省略 → 上一条 +1; 显式 → 直接用。
- **Channel**  (copy):      省略 → 继承; 显式 → 直接用。
- **TickTime** (copy):      省略 → 继承; 显式 → 还要做 nullable 解码 (−1)。

⚠ 如果 Action **继承** (wire 里没有 Action 字节), TickTime 字段会"溜进" Action 后的中间 int 序列前端。解码时要按 Action 期望的中间字段数 (A=3~4, D/C=3, T=5) 反推, 把多出来的首个 int 挪回 prefix。

## 8) 连续 'T' 继承 (同笔成交多对手撮合)

一笔主动单打到多张被动单时, 会产生连续若干条 'T' 记录, 继承 Action, 且其中一边的 ApplSeq 相同 (持续挂在对手盘):

```
rec#9  T 600156  Bid=6000464 Offer=5988780 Price=8.050 Qty=1100 Money=8855
rec#10 T 600156  Bid=6000464 Offer=5993537 Price=8.050 Qty= 600 Money=4830   ← 继承
rec#11 T 600156  Bid=6000464 Offer=5995265 Price=8.050 Qty=1000 Money=8050   ← 继承
rec#12 T 600156  Bid=6000464 Offer=5976233 Price=8.060 Qty=4400 Money=35464  ← 继承
```

同一 BidApplSeq=6000464 代表"主动买单吃掉四张被动卖单", CSV 里这四条是四个连续的 BizIndex。

## 9) 帧切分 & 伪 TID 过滤

`2d ab` 的字节模式 = FAST u32 编码的数字 5803。由于记录里任何 Price/Qty/OrderNO/… 如果数值恰好是 5803, 其字节也是 `2d ab`, 会被纯字节扫描误认为新记录边界。

当前采用的过滤启发式 (见 `9_5803_struct.hpp::TickStreamParser`):
1. `body[i-1]` 必须有 MSB=1 (是 FAST stop 字节, 可能作为下一记录的 PMAP)
2. 和上一个 `tid_pos_` 至少相隔 14B (最短合法记录体)

这把伪命中率从 1.4% 压到 0.02%, 剩下的极少量漂移需要完整的 FAST 字典/模板驱动式解码才能消除。

## 11) Type='S' (TradingPhaseCode / 交易状态) 消息结构

### 概述

全天抓包开头段 (`head5.pcap`) 含有另一种变体记录，对应 CSV 里 `Type='S'` 的行。
这类记录承载的是**集合竞价/停牌/开盘状态字符串**，字段极简，和 'A'/'D'/'T' 交易记录走完全不同的字段布局。

交叉验证结果（`py/validate_head5.py`）：
- 共 3781 条记录（5 条 TCP 流各广播一份，取一份去重后 = 3781）
- `(SecID, Status)` 集合与 CSV 前段 3781 条 Type='S' 行 **完全一致，0 差异**
- 逐行 Status 字段：**3781/3781 = 100% 匹配**

### PMAP 模式及字段布局

观察到三种 PMAP，字段数量依次递增：

| PMAP 字节 | bit 索引（置 1）     | 字段布局                                                    | 记录数 | 说明                   |
|-----------|---------------------|-------------------------------------------------------------|--------|------------------------|
| `48 84`   | [0, 3, 11]          | **SecID** → **TradingPhaseCode**                            | 18785  | 常规状态记录           |
| `4c 84`   | [0, 3, 4, 11]       | **SecID** → **TransactTime** → **TradingPhaseCode**         | 55     | 带时间戳               |
| `7e 84`   | [0, 1, 2, 3, 4, 5, 11] | **SeqA** → **SeqB** → **SecID** → **TransactTime** → **83(常量)** → **TradingPhaseCode** | 65 | 每帧第一条，带帧级序号 |

> 注：以上记录数是对 5 条 TCP 流中单条流的统计，5 条流内容完全相同（广播复制）。

### 字段编码细节

**SecID（6 字节 FAST ASCII）**
- 与 'A'/'D'/'T' 记录相同：前 5 字节 `0x30~0x39`，第 6 字节 `MSB=1`，低 7 位为最后一位数字。

**TradingPhaseCode（变长 FAST ASCII）**
- 末字节 `MSB=1` 终止，低 7 位为末字符。
- 观察到的值：`SUSP`（停牌，占绝大多数）、`OCALL`（集合竞价）、`START`（5 条）。

| 字符串   | 原始字节              |
|----------|-----------------------|
| `SUSP`   | `53 55 53 d0`         |
| `OCALL`  | `4f 43 41 4c cc`      |
| `START`  | `53 54 41 52 d4`      |

**TransactTime（FAST uint，单位：百分秒）**
- 格式：`H * 1,000,000 + MM * 10,000 + SS * 100 + cc`，末两位 cc 为百分之一秒（centiseconds）。
- 例：`9140027` = 09:14:00.27（centiseconds），对应 CSV TickTime `09:14:00.260`（毫秒）。
- **与 CSV 的 TickTime 偏差**：pcap 百分秒 × 10ms 比 CSV 毫秒固定多约 10ms，是 SSE 侧时间戳精度截断的正常结果（22/24 条 Δ = +10ms）。

**SeqA / SeqB（`7e84` 记录的前缀 FAST int）**
- 始终是每帧第一条记录（`rec_idx = 0`）。
- `SeqB` 按帧递增（1, 2, 3 … 20 …）；`SeqA` 变化规律不明，可能是连接/会话标识。
- 常量 `83`（= 0x53 = `'S'`）紧跟在 TransactTime 之后，语义待确认（疑为市场/会话 ID）。

### 帧尾 Trailer 字节问题

早期观察到记录末尾疑似有 `0x48` trailer，**实际是下一条记录 2 字节 PMAP `48 84` 的第一字节**。验证方式：帧内最后一条记录 PMAP 为单字节 `84`（stop-bit 置 1），无 `0x48` 前缀；所有 3781 条记录的 Trailer 字段均为空（`-`）。

### 记录排列顺序

pcap 与 CSV 的 (SecID, Status) 集合完全一致，但 **排列顺序不同**：前 2404 条逐行对齐，从第 2405 条起（全为 OCALL）顺序错位。原因是 pcap 按 TCP 交付顺序排列，CSV 可能经过了按 TickTime 或 SecurityID 的二次排序。
