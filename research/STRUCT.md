# 当前对 `data/output.pcap` 的全部理解（阶段性总结）

## 一、抓包本身

- 文件：`output.pcap`，pcap-ng，54 MB，是一份 77 GB 原始抓包的**中间截段**——意味着**没有 TCP 三次握手、没有登录握手消息**，一上来就是业务流。
- 时间：2026-01-14 09:53:48.78 ~ 09:53:58.75，总时长 **9.97 秒**，50001 个包，约 5000 包/秒、42 Mbps。
- 交易日期：2026-01-14（已由 body 里 `01 35 25 12 = 0x01352512 = 20260114` 交叉验证）。

## 二、网络层

- 5 条并行的 TCP 长连接，全部**单向**从服务端推给客户端：
  - 服务端：`1.4.2.52:5261`、`1.4.2.53:5261`（两台）
  - 客户端：`1.6.86.{11,13,14,16,131}` 的 5 个随机端口
- 每条流内无任何客户端上行业务字节——要么握手在抓包之外，要么集散商按订阅模式**只推不收**。

## 三、传输层帧格式（已完全反推清楚）

每条 TCP 流的 payload 首尾衔接后，**严格按** `0x0004C453` 魔数切帧。5 条流共约 26000 帧，**零偏差**验证通过。40 字节外层包头（**大端**）：

| 偏移  | 大小   | 含义                             | 已验证                             |
| ----- | ------ | -------------------------------- | ---------------------------------- |
| 0–3   | u32    | Magic `0x0004C453`               | 26000 帧全部命中                   |
| 4–7   | u32 BE | 整帧长度（含 40B 头）            | `off[i+1] − off[i] == len[i]` 精确 |
| 8–11  | u32 BE | **type**（通道主类）             | 每流 10–12 种                      |
| 12–15 | u32 BE | **subtype**（通道子类）          | 与 type 一一对应                   |
| 16–19 | u32 BE | **seq**（按 (type,sub) 严格 +1） | 用于丢包检测                       |
| 20–23 | u32    | 恒为 0                           |                                    |
| 24–27 | u32    | 时间戳/相关字段                  | 每帧不同                           |
| 28–31 | u32    | 0 或 `0x08000000`                | 类型相关 flag                      |
| 32–35 | u32    | **压缩标志**：0=明文，1=ZIP      | 与 body 起始 `PK\x03\x04` 一致     |
| 36–39 | u32    | 0/1/0xF（次要 flag）             |                                    |

## 五、业务载荷三大类

1. **SSE 官方 STEP 文本**（4/18/23 通道）：`8=FIXT.1.1|9=长度|35=W|49=EzEI.1.1|56=EzSR.2010600|34=序号|52=时间|347=GBK|…`。与 `sse_step.pdf` 格式一致，`EzEI/EzSR` 的 CompID 揭示了**集散商身份**。
2. **厂商"M201"伪头 BINARY**（48/50）：借用了 SSE BINARY 的 `"M201"` ASCII 标签，但字段布局和官方不同；53B 小包是 status/心跳性质，398B/982B/3462B 大包是多股快照 bundle，内部明确出现 `MDStreamID(5B)+SecurityID(8B)+GBK 名称` 这样的上交所原生字段 layout。
3. **ZIP deflate 压缩的厂商私有增量**（9/47/6-3202/38/39/49）：标准 ZIP Local File Header（`PK\x03\x04 14 00 08 08 08 00`，filename `"0"`，无 Central Directory），raw deflate 解压后是 5 位数字代码 + 停止位编码的短记录，记录尾部统一带 `09 54 4A 93` 魔数。

## 六、解压方法（已跑通）

ZIP 通道流式模式，`zipfile` 模块识别不了。正确做法：

```python
n = struct.unpack('<H', body[26:28])[0]  # filename 长度
m = struct.unpack('<H', body[28:30])[0]  # extra 长度
data = zlib.decompressobj(-zlib.MAX_WBITS).decompress(body[30+n+m:])
```

实测成功解出 10000+ 字节的未压缩载荷。

## 七、已排除的可能性

- **不是 SSE 官方 BINARY/STEP 直连**：有自己的 40B 外层头。
- **不是通联 MDL**：MDL 用 FAST/Protobuf/二进制三选一，没有 `0x0004C453` 魔数和 TCP 多通道混送。
- **不是盛立 EFH**：EFH 是 UDP 组播 + 26B LE `sse_hpf_head`；本 pcap 是 TCP + 40B BE 头，字节序、包头大小、压缩策略都对不上。
- **FIX `EzEI.1.1 → EzSR.2010600` 指向某"Ez-"系集散商**，厂商身份待确认。

## 八、EFH 资料里可以"蹭用"的部分（已挑出）

- `sse_hpf_tick_merge`（72B 合并逐笔定义）——准备用它硬套 (48/50) 的 BINARY 包
- `sse_hpf_exe` / `sse_hpf_order` 字段顺序与精度（价×1000、量×1000、金额×100000）
- `sse_lev2_price_quantity_unit`（16B 档位单元）——用于在 ZIP 解压后 grep 10 档数组
- `sse_static_msg_body` 里 `double` 涨跌停价的 IEEE754 模式，用作解压后记录的定位锚点
- MsgType 枚举 33/34/39/32/38/62/60/61/63/255；SecurityType/SubSecurityType 枚举；TradingPhaseCode 位域——这些枚举和精度**不依赖字节序**，可以直接照抄
